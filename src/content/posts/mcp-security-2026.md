最近 MCP（Model Context Protocol）的生态扩张速度有点猛。Anthropic 去年推出来的时候，大家还在讨论"这东西到底有什么用"，现在回头看，各种 MCP server 已经遍地开花了——操作数据库的、调 API 的、读文件的、搜文档的，基本上你能想到的场景都有人做了。

但有个问题一直被选择性忽略：**安全**。

## 一份不太好看的体检报告

6 月 26 日，一个叫 Capframe 的团队发布了一个 MCP 安全排行榜，用他们自己的规则引擎扫描了 89 个公开的 MCP 服务器。评分标准很简单：满分 100，发现一个 Critical 级别的问题扣 10 分，High 扣 4 分，Medium 扣 2 分，Low 扣 1 分。

结果怎么样？我挑几个大家最熟的说：

- **OpenAI 自己的文档 MCP 服务器**（`developers.openai.com/mcp`）：80 分，B 级。2 个 High，6 个 Medium。
- **AWS Knowledge MCP**：80 分，B 级。3 个 High，4 个 Medium。

对，你没看错，OpenAI 和 AWS 这种大厂的 MCP 服务器都没拿到满分。更别提那些社区做的小工具了。

## 具体是什么问题？

我看了一下 Capframe 的详细报告，最常见的安全问题集中在几个方向：

**1. SSRF 漏洞**

很多 MCP 服务器的工具接受 URL 参数，但没有做任何限制。比如 OpenAI 的 `fetch_openai_doc` 工具，参数里有个 `url` 字段，没有 `pattern` 或 `enum` 约束。这意味着如果有人通过 prompt injection 骗 agent 去访问 `http://169.254.169.254/`（云服务商的元数据地址），就能拿到服务器的内部信息。

这不是理论上的风险。Indirect prompt injection 已经是公认的安全隐患了——攻击者在网页里藏一段指令，agent 读到之后就会执行。

**2. 密钥泄露**

有些 MCP 工具会把 API key、credentials 之类的敏感信息暴露给 agent 的上下文窗口。一旦 agent 的上下文被 prompt injection 攻击者看到，这些密钥就等于白送。Capframe 的报告里，不少工具都踩了这个坑。

正确的做法应该是：密钥在服务端注入，不经过 agent 的上下文。

**3. 输入没有边界**

很多工具的字符串参数没有 `maxLength` 限制，数值参数没有 `maximum` 限制。看起来是个小事，但这给攻击者提供了塞入任意 payload 的空间。

## 我的看法

说句实话，这些问题不意外。

MCP 的设计初衷是让 LLM 能方便地调用外部工具。"方便"和"安全"就是一对天然的矛盾。协议本身提供了一套 tool schema 的规范，但安全完全交给开发者自己负责。这就像给你一把瑞士军刀，但不教你怎么安全使用。

问题在于：现在做 MCP server 的人，很多是"能不能先跑起来"的心态。安全？等产品火了再说。这在创业语境下可以理解，但 MCP 的特殊之处在于，你的工具是被 AI agent 调用的——agent 本身就是一个容易被 prompt injection 攻击的系统。你等于在给一个"容易被骗的执行者"提供高权限的操作接口。

而且 MCP 的使用场景天然涉及敏感操作：读写数据库、调用 API、访问文件系统。这些操作一旦被滥用，出事的概率和影响范围都比普通的 web 漏洞大得多。

## 好消息

也不全是坏消息。排行榜上确实有不少拿到满分 100 的服务器——Cloudflare 的 MCP server、Elastic 的 Elasticsearch MCP server、Playwright MCP server 等，都是 A 级满分。这说明做到安全合规并不需要什么黑科技，基本功扎实就行：

- URL 参数加白名单
- 敏感信息不暴露给 agent 上下文
- 所有输入加上长度/范围约束
- 工具描述里不藏可以被利用的信息

Capframe 还开源了他们的扫描规则（`findings.v1` schema），任何人可以用同样的标准自查。这个做法值得推广。

## 对开发者的建议

如果你在做 MCP server 或者在项目里接入了 MCP：

1. **先跑一遍安全扫描。** Capframe 的工具是开源的，不需要注册就能用。
2. **最小权限原则。** agent 不需要的权限不要给。能只读就不要给写权限。
3. **密钥走服务端。** 永远不要把 credentials 放进 agent 能看到的上下文。
4. **所有输入做约束。** 字符串加长度限制，数字加范围限制，URL 加白名单。

MCP 生态还很年轻，现在养成好习惯比以后修补要便宜得多。

## 写在最后

MCP 是个好协议。它解决了 LLM 工具调用的一个真实痛点——之前每个框架都有自己的 tool 定义方式，互不兼容。MCP 统一了这个层，对整个 agent 生态是好事。

但好协议不等于安全的协议。安全是用出来的，不是设计出来的。Capframe 这份报告来得正是时候——在 MCP 生态大规模铺开之前，先把安全问题摆到台面上。总比出了事再补要好。

希望明年再看这个排行榜的时候，满分的能多几个吧。
