前两天刷 Hacker News 看到一个项目叫 Forge，687 赞。标题挺唬人：让一个 8B 参数的本地小模型在工具调用任务上的准确率从 53% 干到 99%。

我一开始是不信的。

点进去看了下，做的事情其实很朴素。不训练新模型，不搞微调，就是在模型外面包了一层东西。模型调用工具格式写错了，帮你纠正；参数不对，帮你重试；返回结果有问题，帮你验证。就这样。

效果是真的。而且不光小模型受益，连 Claude Sonnet 4.6 都从 85% 被拉到了 98%。

## 不是模型笨，是手不稳

过去一年大家一直在卷模型能力，GPT-4o、Claude 4、Gemini 2.5 Pro，一个比一个猛。但你真的去搭一个能干活的 Agent 试试，最让人崩溃的根本不是模型"想不明白"，是它"做不对"。

比如你让 Agent 查个天气，它知道该调 get_weather 工具，思路没问题。但传参的时候城市名拼错了，或者 JSON 格式不对，工具直接报错。它又试一次，换了个更离谱的格式。来回折腾五六轮，最后跟你说"抱歉，我无法获取天气信息"。

这不是笨。这是手不稳。

Forge 解决的就是这个。核心思路简单到有点无聊：别指望模型每次都完美输出，在模型和工具之间加个纠错层。格式不对？自动修正重试。多步操作？帮你管步骤依赖。上下文太长？帮你压缩。

作者在 README 里说得挺直白：Forge 不是 Agent 编排器，不是代码生成器，就是个可靠性中间件。你现有的架构不用改，往中间一插就行。

## 护栏的两难

Forge 火了没几天，Anthropic 的新模型 Fable 出事了。584 赞，516 条评论，HN 上很少见这么激烈的讨论。

安全研究员把 Fable 拿去做渗透测试、漏洞分析这些正常的网络安全工作，它全拒绝了。不是偶尔拒绝，是基本什么都干不了。

研究员气坏了。你一个号称"最安全"的公司出的模型，连合法的安全研究都做不了？

Anthropic 倒也不是没道理。不限制的话，模型被滥用的风险确实存在。问题是他们的限制太粗暴了，基本就是关键词黑名单，不理解用户到底想干嘛。

Agent 的可靠性到底该谁来保证？Forge 的做法是让应用层保证——模型负责想，中间件负责做对，你想加多严的护栏是你的事。Anthropic 的做法是模型本身就要安全，宁可错杀。

用户不关心你有多"安全"。你的 Agent 连正常任务都完不成，安全就是个笑话。

## 被忽视的烧钱问题

Agent 落地还有个绕不开的事：token 成本。

有人写了篇文章讲 MCP 协议设计不好会导致 Agent 多花 5 倍的 token。听着像技术细枝末节，但想想影响挺大。

Agent 跟普通对话不一样。普通对话一问一答，token 有限。Agent 要反复调工具、处理返回、多轮推理，每次调用都得把工具 schema、历史对话、当前上下文全塞 prompt 里。稍微复杂点的任务，token 轻松破百万。

协议设计烂的话，比如每次都把所有工具定义发给模型（哪怕这次只用一个），返回结果里塞一堆没用的元数据，token 就这么烧掉了。

Coinbase 刚宣布可以让 AI Agent 直接管账户、交易、消费。Agent 能碰真钱了。如果工具调用不可靠导致交易出错，或者 token 成本失控，后果就不是"抱歉我无法完成任务"了。

## 所以呢

现在 Agent 领域有个挺明显的分化。模型厂商在卷能力——更大上下文、更强推理、更多模态。基础设施层在补短板——工具调用要可靠、成本要低、安全边界要清楚。

Forge 2000 多个 star，说明大家被 Agent 不靠谱折磨够了。

接下来半年，我觉得机会不在模型层。谁能搞定三件事——工具调用可靠性、token 成本优化、安全策略精细化——谁就能让 Agent 从"演示很酷"变成"真能干活"。

说白了就是，模型再聪明，手不稳也白搭。
