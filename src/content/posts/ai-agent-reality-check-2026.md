过去两个月发生了一些事，让我对 AI agent 的看法变得更复杂了。

四月份，Forbes 报道说 OpenAI 自己在用 Codex agent 运行数据平台，而且是"自主运行"。六月初，《纽约时报》写了篇长报道，讲美国小企业主怎么管理一整支"AI 员工队伍"。差不多同时，HN 上有人问了个问题："到底有没有人在用 AI agent 做真正的软件？"

大厂说能独立干活了，小企业主在真的用，开发者还在怀疑。我自己也卡在这三种状态之间。

## Agent 确实在干活，但不是你想的那种

先说好的。现在用 AI agent 写代码，体验比两年前强太多了。Claude Code、Codex CLI、Cursor，给一个清晰的需求，它能生成能跑的函数、一套接口、一组测试。以前写半小时的样板代码，现在一句话搞定。

但"能跑"和"能用"之间差着一道坎。

三月份有个报告挺吓人：AI coding agent 生成代码的时候，会无意中引入有漏洞的依赖包。不是故意的——它只是在做统计上"最可能正确"的选择。问题是安全不靠统计。一个有已知 CVE 的包在训练数据里出现过几百次，agent 觉得它"常见"就顺手用了。

Eli Bendersky 写了篇博客讲他用 agent 从零写 Go 项目的经历。他的说法比较实在：agent 能加速编码，但你得一直盯着。有时候一个 changelist 相当于好几天的工作量，但他花了好几个小时让 agent 反复清理和重构。"有提升，只是没某些人吹的那么大。"

## 工程问题，不是 AI 问题

六月中旬有篇文章标题很直白：Why Agents Don't Scale: It's an Engineering Problem, Not an AI Problem。

我看到这个标题愣了一下，然后觉得确实如此。

我们现在碰到的瓶颈，大部分不是模型不够聪明。是围绕 agent 的工程体系没建好。怎么监控一个 agent 在干什么？它做了十个步骤，你怎么知道第七步出了问题？传统软件有 CI/CD、有监控、有告警，agent 系统呢？

最近 HN 上冒出一堆新工具在尝试回答这些问题：V-COS 做 agent 治理层，Sync 做质量控制和项目管理，有人做了"AI coding agent 审计清单"，列了 30 条不变量检查。还有个叫 Ponytail 的项目，目标是"让你的 agent 像团队里最懒的高级工程师一样思考"。这名字我是服气的。

这些工具的出现说明了一件事：agent 的能力已经不是瓶颈了，怎么管 agent 才是。

## Vibe coding 的代价

说到管理，vibe coding 绕不过去。

ACM 六月份发了警告，说 vibe coding（完全不看代码、纯靠 AI 生成的写法）跳过了核心工程实践。从计算机学会嘴里说出来，分量不轻。

Cursor 的 CEO 更早就说过类似的话，说 vibe coding 会"建立摇摇欲坠的基础，最终一切崩塌"。说这话的人是卖 AI 编程工具的，这本身就挺说明问题——他显然看到了什么让他不安的东西。

我自己也碰到过。agent 生成的代码能编译、能过 lint、测试也能跑。但过两周改需求的时候才发现，它用了个奇怪的抽象方式，或者把本该分离的逻辑揉在了一起。"能跑"和"好改"之间差着十万八千里。

所以最近冒出那么多 agent 治理工具不奇怪。不是大家不信任 AI，是发现光信任不够，得有手段验证。

## 小企业和大公司的不同打法

《纽约时报》那篇报道写的都是美国小企业主，用 OpenClaw 之类的平台雇了一群 AI agent 当员工——处理邮件、做报表、管库存。对他们来说不需要 agent 多聪明，能把重复劳动干了就行。

这跟 OpenAI 自己用 Codex 管数据平台是完全不同的需求。大公司要的是 agent 在复杂系统里自主决策，小企业要的是 agent 别添乱、把琐碎事干了。

需求不同，打法不同。但核心问题是一样的：agent 能不能在特定场景下稳定交付结果。

## 所以呢

AI agent 已经过了"能不能用"的阶段，现在卡在"怎么用好"上。模型能力还会涨，但决定 agent 能走多远的不是模型，是围绕它的工程体系——监控、治理、安全、测试。

最近看到一句话：Nobody Is Measuring What Your AI Agents Are Worth。说得挺准的。我们都在用 agent，但很少有人认真算过它带来了多少价值、引入了多少风险。

如果你要在正式项目里用 AI agent，我的建议很简单：先想清楚怎么验证它的输出，再想清楚怎么用它。不是 agent 不行，是我们管 agent 的方法还远远没跟上。
